## Live baseline finding

- `https://okusuri-mimamori.com/.well-known/assetlinks.json` redirects to `www`.
- `https://www.okusuri-mimamori.com/.well-known/assetlinks.json` currently returns HTTP 404.

Therefore the Android `autoVerify=true` intent filter is implemented but production App Links are not yet verified.

## Implemented server contract

The API now serves `/.well-known/assetlinks.json` from `ANDROID_APP_LINK_SHA256_CERT_FINGERPRINTS`. It accepts comma-separated SHA-256 certificate fingerprints, uppercases and deduplicates them, rejects malformed entries and emits the fixed package `com.afterlifearchive.medmanager` with relation `delegate_permission/common.handle_all_urls`.

Missing or entirely invalid configuration returns HTTP 503 with `Cache-Control: no-store`; it never publishes an empty or guessed trust association. Valid output is JSON with a five-minute public cache. Integration tests cover fail-closed behavior, validation/normalization/deduplication and the complete response contract.

## Production completion steps

1. Enable Play App Signing and obtain the **App signing key certificate SHA-256** from Play Console. Add the upload-certificate SHA-256 only if a directly installed upload-signed build must also verify.
2. Set `ANDROID_APP_LINK_SHA256_CERT_FINGERPRINTS` in the production Vercel environment and deploy this exact API commit.
3. Confirm the final `www` URL returns HTTP 200 JSON.
4. Install the Play-delivered build, run `adb shell pm verify-app-links --re-verify com.afterlifearchive.medmanager`, and inspect `adb shell pm get-app-links com.afterlifearchive.medmanager`.
5. Open both production `/auth/...` links externally and confirm direct Android routing, browser fallback when absent, and no custom-scheme dependency.

The production signing certificate and deploy authority are not available locally, so this remains `PARTIAL` rather than `VERIFIED`.
